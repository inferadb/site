Every multi-tenant authorization system makes the same promise: your data is isolated from other tenants. The question is how that promise is enforced. In most systems — including every Zanzibar-derived engine we are aware of — tenant isolation is implemented at the application level. A tenant identifier is attached to each query, and the query layer filters results to include only data belonging to that tenant. The storage layer has no awareness of tenant boundaries. All tenants share the same tables, the same indexes, the same encryption keys, and the same physical storage. The isolation guarantee depends entirely on the correctness of the application-level filtering logic. A single bug in a query path, a missing filter in a new endpoint, or a SQL injection that bypasses the tenant filter, and data leaks across tenant boundaries.

InferaDB enforces tenant isolation at the storage engine level through a mechanism we call vaults. Each vault is an independent unit of encryption, storage, and consensus. Every key in the B+ tree is prefixed with the vault's 8-byte identifier, which means that data belonging to different vaults is structurally separated in the key space. But the real isolation comes from encryption: each vault has its own independent AES-256-GCM envelope encryption hierarchy. The vault's data encryption key (DEK) encrypts every page of data belonging to that vault. The DEK is itself encrypted by a key encryption key (KEK) that is unique to the vault. A query executing in the context of vault A literally cannot decrypt the pages belonging to vault B, even if a bug in the query layer caused it to read the wrong pages. Cross-tenant data access is not merely prevented by policy — it is made architecturally impossible by the encryption boundary.

Each vault maintains its own independent blockchain — a chain of blocks where each block contains a batch of committed operations and a state root derived from a Merkle tree over the vault's current state. The state root is a cryptographic commitment to the entire contents of the vault at that point in time. Any modification to any piece of data within the vault changes the state root, which means that tampering with the audit trail is detectable. The per-vault blockchain is independent from other vaults' chains, which means that the audit history of one tenant cannot be influenced by operations on another tenant. An auditor verifying the integrity of tenant A's access history does not need to process or even be aware of tenant B's data.

This architecture enables cryptographic shredding, which is the most robust mechanism for complying with GDPR Article 17's right to erasure. When a tenant requests that their data be deleted, InferaDB destroys the vault's key encryption key. Without the KEK, the DEK cannot be recovered. Without the DEK, the encrypted pages are indistinguishable from random bytes. The data is irrecoverable by any party, including InferaDB's operators. This is a stronger guarantee than row-level deletion in a shared database, where deleted data may persist in backups, write-ahead logs, or filesystem slack space. Cryptographic shredding is instantaneous — destroying a key is a constant-time operation regardless of how much data the vault contains — and it is provably complete, because the encryption is the only means of accessing the data.

Data residency requirements add another dimension to tenant isolation. Regulations like GDPR, DORA, and various national data sovereignty laws require that certain categories of data be stored within specific geographic boundaries. InferaDB supports this through region-pinned Raft groups: a vault's Raft group can be configured to include only nodes within a specific region, ensuring that all replicas of the vault's data remain within the required jurisdiction. This is enforcement at the infrastructure level, not a policy annotation that depends on operational procedures for compliance. The vault's data physically cannot leave the designated region because the consensus protocol does not replicate it to nodes outside that region. Combined with per-vault encryption, this means that each tenant's data is both cryptographically isolated and geographically constrained, with both guarantees enforced by the system architecture rather than by application code.
