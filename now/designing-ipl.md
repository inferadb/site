Most authorization systems force developers to choose a paradigm. Relationship-based access control systems like Zanzibar excel at modeling organizational hierarchies and sharing relationships but struggle with attribute-based conditions. Policy engines like OPA handle arbitrary logic but require developers to write and maintain imperative code for every access decision. Role-based systems are simple to understand but collapse under the weight of role explosion in complex organizations. The Infera Policy Language was designed to unify these paradigms into a single schema language that can express relationship-based, role-based, and attribute-based authorization rules in a coherent, analyzable syntax.

IPL is defined by a PEG (Parsing Expression Grammar), which gives it unambiguous parsing semantics — there is exactly one way to parse any valid IPL document, which eliminates an entire class of subtle bugs that arise from ambiguous grammars. The core keywords are deliberately few: `type` defines an entity type, `relation` defines a relationship, `forbid` declares an explicit deny rule, `this` refers to the current type, and `from` expresses computed usersets that traverse relationships. The expression operators compose these primitives: union (`|`) combines permissions from multiple sources, intersection (`&`) requires all conditions to hold, exclusion (`-`) removes specific grants, computed userset (`from`) follows relationship chains, related object (`->`) traverses typed relationships, and `module()` invokes a WebAssembly policy module for domain-specific logic.

The evaluation semantics follow a strict precedence: forbid rules are always evaluated before permit rules, implementing an explicit deny model. If any forbid rule matches, access is denied regardless of what permit rules might also match. This is a deliberate design choice that makes security boundaries unambiguous. A common pattern in enterprise authorization is "all editors can modify documents, except documents in the legal hold collection." In IPL, this is expressed as a forbid rule on the legal hold type that takes precedence over the editor permission, regardless of how the editor relationship was derived. Developers do not need to reason about rule ordering or worry that a broadly-scoped permit will override a narrowly-scoped restriction.

IPL documents pass through three validation phases before they are accepted by the system. The first phase is type checking, which verifies that all referenced types and relations exist, that relationship expressions connect compatible types, and that module references point to uploaded WASM modules. The second phase is conflict detection, which identifies rules that can never match (dead code), rules that always overlap (redundant grants), and forbid rules that shadow permit rules in ways that might not be intentional. The third phase is coverage analysis, which reports which type-action pairs have no rules defined, helping developers identify gaps in their authorization schema before they become security vulnerabilities. These three phases catch the majority of authorization bugs at schema deployment time rather than at runtime.

The query engine evaluates IPL expressions as a directed acyclic graph, with each node representing a sub-expression and edges representing data dependencies. Independent branches — for example, the two sides of a union expression — are evaluated in parallel. The engine performs query cost estimation before execution, using statistics about the cardinality of relationship sets to choose between materialization (pre-computing all members of a set) and iteration (checking membership on demand). For large relationship graphs, this cost-based optimization can mean the difference between a microsecond response and a millisecond response. Cycle detection runs at schema validation time, rejecting any type definition that would create an infinite traversal loop during evaluation. The combination of static analysis at deploy time and cost-based optimization at query time means that IPL schemas are both safe and performant by construction.
