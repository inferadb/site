Retrieval-augmented generation has become the standard architecture for connecting large language models to enterprise data. The premise is simple: instead of fine-tuning a model on proprietary data, you retrieve relevant documents at query time and include them in the model's context window. The model generates a response grounded in real, up-to-date information rather than relying solely on its training data. The security problem is equally simple: without authorization controls on the retrieval step, the model can access any document in the corpus. When a user asks a question, the retrieval system returns the most semantically relevant documents regardless of whether that user is authorized to see them. The model then generates a response that may incorporate confidential information the user should never have been exposed to.

The most common approach to this problem is post-retrieval filtering. The retrieval system fetches the top-k most relevant documents, and then a filter removes any documents the user is not authorized to access. This approach is fundamentally insecure for a critical reason: the unauthorized documents have already been retrieved. In systems that use embedding-based retrieval, the document content is encoded in the embeddings, and the retrieval step itself may log or cache the full document text. In systems where the retrieval and filtering happen in the same process, a bug in the filtering logic — a missing check, an exception that bypasses the filter — results in unauthorized documents reaching the model. Post-retrieval filtering treats authorization as a convenience layer rather than a security boundary, and the consequences of a filter bypass are severe: the model generates a response containing information it should never have seen, and that response is delivered to the user.

InferaDB enables a fundamentally different approach: pre-retrieval authorization at the database level. Before the retrieval system queries the vector store or document index, it checks authorization for each candidate document against the requesting user. The authorization model represents document access as relationship tuples — `user:alice` has relation `viewer` on `document:quarterly-report` — which can be evaluated in bulk. Because InferaDB achieves 2.8 microsecond p99 read latency, checking authorization for a thousand candidate documents adds less than 3 milliseconds to the retrieval pipeline. This is negligible compared to the embedding search and model inference steps that dominate RAG latency. Unauthorized documents are excluded before they are retrieved, before their content enters any cache or log, and before the model can incorporate them into a response.

The authorization model for RAG pipelines maps naturally to InferaDB's relationship-based schema. Organizations model their document hierarchy — departments, projects, teams, shared folders — as typed relationships in the Infera Policy Language. A user's access to a document is derived from their membership in teams, their role in projects, and any direct sharing relationships. When a new document is ingested into the RAG pipeline, a single relationship tuple is written to InferaDB that places the document in the appropriate location in the hierarchy. The document inherits the access policies of its parent container. When an employee changes teams, updating their team membership in InferaDB automatically updates their access to all documents associated with that team. There is no per-document access control list to maintain, and the authorization model stays in sync with organizational structure without batch synchronization jobs.

Auditability is the other half of the RAG security equation. When an AI agent generates a response, it is essential to know what documents were accessed, which user the agent was acting on behalf of, and what authorization decisions were made. InferaDB records every authorization check in its tamper-evident audit trail, producing a chain-of-thought permission trace for each agent interaction. This trace shows, for each document that the retrieval system considered, whether access was granted or denied and why — which relationship path was traversed, which policy rule matched, and at what revision of the authorization state the decision was made. For regulated industries where AI-generated outputs may inform consequential decisions, this trace provides the evidence that the system operated within its authorized boundaries. It transforms the authorization layer from a silent gatekeeper into a documented participant in the AI pipeline's decision chain.
