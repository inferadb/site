The decision to build InferaDB in Rust was not driven by language preference. It was driven by a set of non-negotiable performance requirements that eliminated most alternatives. An authorization database must deliver sub-microsecond read latency at the storage engine level, sustain hundreds of thousands of concurrent operations, and do both without the latency spikes that garbage-collected runtimes introduce. Go, Java, and C# all have mature ecosystems for distributed systems, but their garbage collectors impose pause times that are measured in milliseconds — acceptable for most workloads, but incompatible with a system where the entire read path must complete in under 3 microseconds at p99. C and C++ offer the necessary performance characteristics, but their manual memory management introduces the class of security vulnerabilities — buffer overflows, use-after-free, dangling pointers — that an authorization system cannot afford. Rust's ownership model provides memory safety without runtime overhead, which is the specific combination our requirements demanded.

The ownership and borrowing system delivers a second critical benefit beyond memory safety: fearless concurrency. Authorization evaluation frequently involves traversing a graph of relationships — following chains of group memberships, role assignments, and resource hierarchies to determine whether a path exists from a subject to a resource. These traversals have natural parallelism: independent branches of the graph can be explored concurrently. In languages with garbage collectors or runtime-managed memory, concurrent data access requires careful synchronization that is difficult to get right and expensive when you do. Rust's type system enforces data race freedom at compile time. If the code compiles, concurrent access to shared data structures is safe. This guarantee allowed us to implement parallel graph traversal in the authorization evaluator without the defensive locking strategies that dominate concurrent code in other languages.

Rust's zero-cost abstractions are essential for hot-path optimization. The B+ tree storage engine operates on fixed-size pages, performs byte-level key comparisons, and manages memory layouts that must be precisely controlled for cache efficiency. In a higher-level language, these operations would involve boxing, dynamic dispatch, or runtime type checks that add nanoseconds per operation — nanoseconds that compound across the thousands of operations in a single authorization evaluation. Rust's generics are monomorphized at compile time, trait methods are statically dispatched when the concrete type is known, and the compiler aggressively inlines small functions. The abstraction boundaries that make the code maintainable — separating the page cache from the tree traversal from the key encoding — have zero runtime cost. The compiled binary performs as if every layer were hand-written as a single monolithic function.

The async ecosystem, anchored by the Tokio runtime, handles the I/O-intensive portions of the system: Raft message passing, client request handling, and inter-node communication. Axum serves the HTTP API, Tonic handles gRPC for internal cluster communication, and both run on the Tokio runtime with work-stealing scheduling. The Moka caching library provides a concurrent, bounded cache with TTL-based eviction that we use for the authorization evaluation cache layer. OpenRaft gives us a correct, well-tested Raft consensus implementation that we adapted for our specific state machine and log storage. The Wasmtime crate — itself written in Rust — provides the WebAssembly sandbox for custom policy modules, with the Cranelift JIT compiler generating native code from WASM bytecode. Each of these dependencies was chosen because it is production-hardened and because Rust's package ecosystem makes composition safe: type mismatches, lifetime violations, and thread safety issues are caught at compile time rather than in production.

The trade-off is real: Rust has a smaller hiring pool than Go or Java, and the learning curve for developers new to ownership semantics is steep. We accepted this trade-off because the performance characteristics of InferaDB are not incremental improvements that could be achieved by optimizing code in a higher-level language. The difference between 2.8 microseconds and 5 milliseconds is architectural. It comes from eliminating garbage collection pauses, from lock-free concurrent reads, from zero-cost abstractions over the storage engine, and from compile-time guarantees that eliminate the runtime checks other languages require for safety. These are properties of the language and runtime, not the application code. Choosing a different language would have meant accepting fundamentally different performance characteristics, which would have meant building a fundamentally different product.
