Authorization is the most critical security boundary in modern software, and it is failing at scale. Broken access control has held the number one position on the OWASP Top 10 since 2021, affecting 94% of audited applications. The average cost of a data breach now exceeds $4.88 million, and the root cause is almost always the same: an authorization check that was missing, stale, or wrong. Despite this, the infrastructure that developers rely on to enforce access control has barely evolved in decades. Most systems still treat authorization as an afterthought — a middleware layer bolted onto application code, backed by a general-purpose database that was never designed for the job.

We know this problem intimately because we helped build the systems that defined the current generation of authorization infrastructure. Members of our team were core contributors to OpenFGA at Auth0 and continued that work through the Okta acquisition. We shipped a Zanzibar-inspired relationship-based access control system that thousands of developers adopted. We saw firsthand what worked and, more importantly, what didn't. OpenFGA and systems like it represent a meaningful step forward in authorization modeling, but they all share a fundamental architectural limitation: they are policy engines, not databases. They depend on PostgreSQL or MySQL for storage, require external caching layers to achieve acceptable latency, and implement multi-tenancy at the application level rather than the storage level. These are not minor implementation details. They are architectural constraints that impose hard ceilings on performance, security, and operational simplicity.

The reliance on general-purpose databases means that every authorization check pays the cost of query parsing, transaction isolation, and storage abstractions designed for workloads that look nothing like permission evaluation. A simple "can user X access resource Y" check that should take microseconds instead takes 5 to 50 milliseconds when it hits PostgreSQL. Caching helps, but it introduces its own class of problems — stale permissions, cache invalidation bugs, and the fundamental tension between consistency and performance that Google's Zanzibar paper identified as the "new enemy problem." Application-level multi-tenancy means that a single bug in tenant isolation logic can expose one customer's permissions to another. These are not theoretical risks. They are the failure modes we observed repeatedly in production.

InferaDB is our answer: a purpose-built authorization database designed from first principles for the permission evaluation workload. We wrote a custom B+ tree storage engine in Rust that achieves 2.8 microsecond p99 read latency and sustains over 952,000 operations per second. We use Raft consensus for linearizable writes, ensuring that when a permission is revoked, no replica can serve a stale grant. Each tenant operates within a cryptographically isolated vault with independent AES-256-GCM envelope encryption and its own blockchain for tamper-evident audit logging. We built a WebAssembly sandbox for custom policy logic, allowing developers to extend authorization with domain-specific rules written in any language that compiles to WASM, without sacrificing the security guarantees of the core system.

This is not another policy engine with a new query language. It is a new category of infrastructure: a database purpose-built for authorization. The distinction matters because the problems we are solving — sub-microsecond latency, cryptographic tenant isolation, linearizable consistency, tamper-evident audit trails — cannot be retrofitted onto a system that delegates storage to PostgreSQL. They require control over every layer of the stack, from the page layout on disk to the consensus protocol that replicates writes across nodes. That is what we are building, and we believe it represents the next meaningful advance in how software systems enforce access control.
